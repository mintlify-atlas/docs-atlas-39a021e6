## Overview

Deploying Justina Core Backend to production requires careful consideration of security, performance, scalability, and monitoring. This guide covers best practices and considerations for production deployments.

## Pre-Deployment Checklist

<Steps>
  <Step title="Security Hardening">
    - [ ] Generate cryptographically secure JWT secret (min 256 bits)
    - [ ] Use strong database passwords
    - [ ] Enable HTTPS/TLS for all connections
    - [ ] Configure CORS for specific frontend origins only
    - [ ] Change default user credentials
    - [ ] Disable SQL logging (`SPRING_JPA_SHOW_SQL=false`)
    - [ ] Review and update security headers
  </Step>

  <Step title="Database Configuration">
    - [ ] Migrate from H2 to PostgreSQL
    - [ ] Set up database backups
    - [ ] Configure connection pooling
    - [ ] Set `SPRING_JPA_HIBERNATE_DDL_AUTO=update` or `validate`
    - [ ] Create database indexes for performance
    - [ ] Enable PostgreSQL SSL connections
  </Step>

  <Step title="Environment Variables">
    - [ ] All required variables configured
    - [ ] Secrets stored securely (not in plain text)
    - [ ] Environment-specific values verified
    - [ ] No development/test values in production config
  </Step>

  <Step title="Monitoring & Logging">
    - [ ] Configure centralized logging
    - [ ] Set up health check endpoints
    - [ ] Enable application metrics
    - [ ] Configure alerting for critical errors
    - [ ] Set up database monitoring
  </Step>

  <Step title="Performance">
    - [ ] Configure JVM memory settings
    - [ ] Enable JVM performance flags
    - [ ] Set up CDN for static assets
    - [ ] Configure reverse proxy (Nginx/Apache)
    - [ ] Enable HTTP/2
  </Step>

  <Step title="High Availability">
    - [ ] Multiple application instances
    - [ ] Load balancer configured
    - [ ] Database replication/clustering
    - [ ] Automatic failover configured
    - [ ] Health checks on load balancer
  </Step>
</Steps>

## Database Migration: H2 to PostgreSQL

### Why PostgreSQL?

| Feature | H2 (Development) | PostgreSQL (Production) |
|---------|------------------|-------------------------|
| **Persistence** | In-memory (data lost on restart) | Persistent storage |
| **Performance** | Limited for concurrent users | Optimized for high load |
| **Reliability** | Not production-ready | Battle-tested |
| **Features** | Basic SQL support | Advanced features (JSON, full-text search) |
| **Scalability** | Single process | Replication, clustering |

<Warning>
  **Never use H2 in production!** All data will be lost when the application restarts.
</Warning>

### PostgreSQL Setup

<Steps>
  <Step title="Install PostgreSQL">
    <CodeGroup>

    ```bash Ubuntu/Debian
    sudo apt update
    sudo apt install postgresql postgresql-contrib
    sudo systemctl start postgresql
    sudo systemctl enable postgresql
    ```

    ```bash RHEL/CentOS
    sudo dnf install postgresql-server postgresql-contrib
    sudo postgresql-setup --initdb
    sudo systemctl start postgresql
    sudo systemctl enable postgresql
    ```

    ```bash macOS
    brew install postgresql@15
    brew services start postgresql@15
    ```

    ```bash Docker
    docker run -d \
      --name justina-postgres \
      -e POSTGRES_DB=justina \
      -e POSTGRES_USER=justina_user \
      -e POSTGRES_PASSWORD=secure_password \
      -v postgres-data:/var/lib/postgresql/data \
      -p 5432:5432 \
      postgres:15-alpine
    ```

    </CodeGroup>
  </Step>

  <Step title="Create Database and User">
    ```sql
    -- Connect to PostgreSQL
    sudo -u postgres psql

    -- Create database
    CREATE DATABASE justina;

    -- Create user with password
    CREATE USER justina_user WITH ENCRYPTED PASSWORD 'your_secure_password';

    -- Grant privileges
    GRANT ALL PRIVILEGES ON DATABASE justina TO justina_user;

    -- Switch to justina database
    \c justina

    -- Grant schema privileges
    GRANT ALL ON SCHEMA public TO justina_user;

    -- Exit
    \q
    ```
  </Step>

  <Step title="Update Environment Variables">
    Update your `.env` file or environment configuration:

    ```bash
    # Database (PostgreSQL)
    SPRING_DATASOURCE_URL="jdbc:postgresql://localhost:5432/justina"
    SPRING_DATASOURCE_USERNAME="justina_user"
    SPRING_DATASOURCE_PASSWORD="your_secure_password"
    SPRING_DATASOURCE_DRIVER_CLASS_NAME="org.postgresql.Driver"

    # JPA/Hibernate
    SPRING_JPA_HIBERNATE_DDL_AUTO="update"
    SPRING_JPA_SHOW_SQL="false"
    SPRING_JPA_PROPERTIES_HIBERNATE_DIALECT="org.hibernate.dialect.PostgreSQLDialect"
    ```
  </Step>

  <Step title="Verify Connection">
    ```bash
    # Test PostgreSQL connection
    psql -h localhost -U justina_user -d justina -c "SELECT version();"
    ```
  </Step>

  <Step title="Start Application">
    ```bash
    # The application will automatically create tables on first startup
    ./mvnw spring-boot:run
    ```

    Verify in logs:
    ```log
    Hibernate: create table users (id bigint generated by default as identity, ...)
    Hibernate: create table surgery_session (id uuid not null, ...)
    ```
  </Step>
</Steps>


### Docker Compose Production Deployment

<CodeGroup>

```yaml docker-compose.prod.yml
version: '3.8'

services:
  justina-backend:
    image: justina-backend:${VERSION:-latest}
    container_name: justina-backend
    restart: always
    ports:
      - "8080:8080"
    environment:
      - PORT=8080
      - JWT_SECRET_KEY=${JWT_SECRET_KEY}
      - SPRING_DATASOURCE_URL=jdbc:postgresql://postgres:5432/justina
      - SPRING_DATASOURCE_USERNAME=${DB_USERNAME}
      - SPRING_DATASOURCE_PASSWORD=${DB_PASSWORD}
      - SPRING_DATASOURCE_DRIVER_CLASS_NAME=org.postgresql.Driver
      - SPRING_JPA_HIBERNATE_DDL_AUTO=update
      - SPRING_JPA_SHOW_SQL=false
      - SPRING_JPA_PROPERTIES_HIBERNATE_DIALECT=org.hibernate.dialect.PostgreSQLDialect
    depends_on:
      postgres:
        condition: service_healthy
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/actuator/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 60s
    networks:
      - justina-network
    logging:
      driver: "json-file"
      options:
        max-size: "10m"
        max-file: "3"

  postgres:
    image: postgres:15-alpine
    container_name: justina-postgres
    restart: always
    environment:
      - POSTGRES_DB=justina
      - POSTGRES_USER=${DB_USERNAME}
      - POSTGRES_PASSWORD=${DB_PASSWORD}
    volumes:
      - postgres-data:/var/lib/postgresql/data
      - ./init-scripts:/docker-entrypoint-initdb.d
    ports:
      - "5432:5432"
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${DB_USERNAME}"]
      interval: 10s
      timeout: 5s
      retries: 5
    networks:
      - justina-network
    logging:
      driver: "json-file"
      options:
        max-size: "10m"
        max-file: "3"

  # Optional: Nginx reverse proxy
  nginx:
    image: nginx:alpine
    container_name: justina-nginx
    restart: always
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
      - ./ssl:/etc/nginx/ssl:ro
    depends_on:
      - justina-backend
    networks:
      - justina-network

volumes:
  postgres-data:
    driver: local

networks:
  justina-network:
    driver: bridge
```

```nginx nginx.conf
upstream justina_backend {
    server justina-backend:8080;
}

server {
    listen 80;
    server_name yourdomain.com;
    
    # Redirect HTTP to HTTPS
    return 301 https://$server_name$request_uri;
}

server {
    listen 443 ssl http2;
    server_name yourdomain.com;

    # SSL Configuration
    ssl_certificate /etc/nginx/ssl/cert.pem;
    ssl_certificate_key /etc/nginx/ssl/key.pem;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;

    # Security Headers
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-XSS-Protection "1; mode=block" always;

    # API Proxy
    location /api/ {
        proxy_pass http://justina_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # WebSocket Proxy
    location /ws/ {
        proxy_pass http://justina_backend;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 86400;
    }

    # Swagger UI
    location /swagger-ui/ {
        proxy_pass http://justina_backend;
        proxy_set_header Host $host;
    }
}
```

## Default Credentials & Security

### Default Users

The application initializes with these default users on first startup:

| Username | Password | Role | Purpose |
|----------|----------|------|----------|
| `surgeon_master` | `justina2024` | `ROLE_SURGEON` | Default surgeon account |
| `ia_justina` | `ia_secret_2024` | `ROLE_IA` | AI system integration account |

<Warning>
  **CRITICAL: Change these credentials immediately in production!**
  
  Default credentials are a major security vulnerability. Attackers often target default credentials.
</Warning>

### Changing Default Credentials

<Steps>
  <Step title="Via API After First Deploy">
    ```bash
    # Login with default credentials
    curl -X POST https://yourdomain.com/api/v1/auth/login \
      -H "Content-Type: application/json" \
      -d '{
        "username": "surgeon_master",
        "password": "justina2024"
      }'

    # Response includes JWT token
    # Use token to update password via user management endpoint
    ```
  </Step>

  <Step title="Via Database (PostgreSQL)">
    ```sql
    -- Connect to database
    psql -U justina_user -d justina

    -- Update password (BCrypt hash)
    -- Generate hash: https://bcrypt-generator.com/
    UPDATE users 
    SET password = '$2a$10$your-bcrypt-hashed-password-here'
    WHERE username = 'surgeon_master';

    UPDATE users 
    SET password = '$2a$10$your-bcrypt-hashed-password-here'
    WHERE username = 'ia_justina';
    ```
  </Step>

  <Step title="Disable Default Users">
    Consider creating new admin users and disabling defaults:

    ```sql
    -- Disable default users
    UPDATE users SET enabled = false WHERE username = 'surgeon_master';
    UPDATE users SET enabled = false WHERE username = 'ia_justina';
    ```
  </Step>
</Steps>

### Security Best Practices

<CardGroup cols={2}>
  <Card title="Passwords" icon="key">
    - Minimum 12 characters
    - Mix of uppercase, lowercase, numbers, symbols
    - No common words or patterns
    - Use password manager
    - Rotate regularly (90 days)
  </Card>

  <Card title="JWT Secrets" icon="shield">
    - Minimum 256 bits (32+ characters)
    - Cryptographically random
    - Never commit to version control
    - Store in secret manager
    - Rotate periodically
  </Card>

  <Card title="Database" icon="database">
    - Unique credentials per environment
    - Least privilege principle
    - Enable SSL/TLS connections
    - Network isolation
    - Regular backups
  </Card>

  <Card title="Access Control" icon="lock">
    - Role-based access (RBAC)
    - Audit logging
    - Session timeouts
    - Multi-factor authentication (future)
    - IP whitelisting for admin
  </Card>
</CardGroup>

## Performance Optimization

### JVM Configuration

Optimize Java runtime for production:

```dockerfile
# In Dockerfile
ENTRYPOINT ["java", \
  "-XX:+UseG1GC", \
  "-XX:MaxGCPauseMillis=200", \
  "-XX:+UseStringDeduplication", \
  "-Xms512m", \
  "-Xmx2g", \
  "-jar", "app.jar"]
```

### Database Indexing

Create indexes for frequently queried columns:

```sql
-- Index on surgery_session user_id (foreign key)
CREATE INDEX idx_surgery_session_user_id ON surgery_session(user_id);

-- Index on surgery_session timestamp
CREATE INDEX idx_surgery_session_timestamp ON surgery_session(created_at);

-- Index on movements surgery_session_id
CREATE INDEX idx_movements_session_id ON movements(surgery_session_id);
```

## Backup & Disaster Recovery

### Database Backups

<CodeGroup>

```bash Automated Backup Script
#!/bin/bash
# backup-postgres.sh

BACKUP_DIR="/backups/postgres"
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="$BACKUP_DIR/justina_$DATE.sql"

# Create backup directory
mkdir -p $BACKUP_DIR

# Dump database
pg_dump -U justina_user -d justina > $BACKUP_FILE

# Compress
gzip $BACKUP_FILE

# Delete backups older than 30 days
find $BACKUP_DIR -name "*.sql.gz" -mtime +30 -delete

echo "Backup completed: $BACKUP_FILE.gz"
```

```bash Cron Schedule
# Add to crontab: crontab -e
# Daily backup at 2 AM
0 2 * * * /usr/local/bin/backup-postgres.sh
```

```bash Docker Volume Backup
# Backup Docker volume
docker run --rm \
  -v justina_postgres-data:/data \
  -v $(pwd)/backups:/backup \
  alpine tar czf /backup/postgres-data-$(date +%Y%m%d).tar.gz /data
```

</CodeGroup>

### Restore from Backup

```bash
# Restore PostgreSQL dump
gunzip -c justina_20240115_020000.sql.gz | psql -U justina_user -d justina

# Restore Docker volume
docker run --rm \
  -v justina_postgres-data:/data \
  -v $(pwd)/backups:/backup \
  alpine sh -c "cd /data && tar xzf /backup/postgres-data-20240115.tar.gz --strip 1"
```

## Scaling Considerations

### Horizontal Scaling

Run multiple backend instances behind a load balancer:

```yaml
services:
  justina-backend:
    # ...
    deploy:
      replicas: 3
    
  load-balancer:
    image: nginx:alpine
    # ... load balancer config
```

### Database Replication

Set up PostgreSQL primary-replica replication for read scaling and high availability.

### Session Management

JWT tokens are stateless, enabling seamless horizontal scaling without shared session storage.

## Troubleshooting

### Application Won't Start

1. Check environment variables are set
2. Verify database is reachable
3. Check logs: `docker-compose logs justina-backend`
4. Verify port 8080 is not in use: `netstat -tuln | grep 8080`

### Database Connection Issues

```bash
# Test PostgreSQL connection from container
docker-compose exec justina-backend sh
apk add postgresql-client
psql -h postgres -U justina_user -d justina
```

### Performance Issues

1. Check database query performance
2. Review application logs for slow queries
3. Monitor JVM memory usage
4. Analyze connection pool metrics
5. Check for N+1 query problems
